Setup

You can download the test script (io_test) and the io samples (io.zip) from the shared OneDrive folder. Move the test script to your projects folder, and extract the io archive to a folder named io in your projects folder. You can do this with the following commands:

# These commands assume io.zip and io_test are in your downloads folder
# The cd command should use the actual path to your projects folder
cd path/to/your/projects/folder
cp ~/Downloads/io_test ./io_test
unzip ~/Downloads/io.zip

If you're using the Safari browser on a Mac, then it's possible your browser already extracted the zip when you downloaded it. In that case, you can just move the extracted io folder to your projects folder.

The test script and extracted io folder should now reside in your projects folder along with all of your programs. If you run the ls command in your projects folder, then you should see the following:

CharacterSheet1.java
CharacterSheet2.java
CharacterSheet3.java
Dice.java
DiceGame1.java
DiceGame2.java
DiceGame3.java
io
io_test

Some of the programs may be missing from that list if you haven't yet started them, but all of the programs you are working on should be in this folder. If you run ls io, then you should see folders numbered 1 through 6. These contain the input and output samples for each project.

If ls io shows another folder named io, then try ls io/io. If that shows numbers from 1 to 6, then your io folder is inside of a second folder named io. You can fix that with mv io/io/* io/ or by manually moving those folders from the inner io folder to the outer io folder. Run rmdir io/io after you move the numbered folders to delete the empty extra io folder. This command is safe and will only work on an empty folder.

Finally, you should make the io_test script executable so you can run it more easily:

chmod u+x io_test

Test Version of Dice.java

When testing Projects 3, 4, 5, and 6, you will need a modified version of Dice.java and an extra line of code at the start of your main function. Add the following line to the beginning of your main function:

Dice.init();

Then replace the Dice.java used by your projects with the following version:

import java.util.Random;
import java.io.IO;

/**
 * Modified version of the Dice class for use during testing. To make your
 * code from Projects 3+ compatible with the test script, add the following
 * line to the start of your main function:
 *
 * Dice.init();
 *
 * This will read the first input as the seed for a random number generator and
 * ensure your program receives the same series of "random" dice rolls as the
 * code that produced the expected output for that I/O sample.
 *
 * @author Zach Kohlberg
 */
@SuppressWarnings("preview")
public class Dice {
    private static Random rand = null;

    public static void init() {
        if (rand == null) {
            try {
                rand = new Random(Long.parseLong(IO.readln("")));
            } catch (NumberFormatException e) {
                rand = new Random();
            }
        } else {
            throw new RuntimeException("You cannot init Dice more than once!");
        }
    }

    public static int roll(int sides) {
        return rand.nextInt(sides) + 1;
    }

    public static int d4() {
        return roll(4);
    }

    public static int d6() {
        return roll(6);
    }

    public static int d8() {
        return roll(8);
    }

    public static int d10() {
        return roll(10);
    }

    public static int d12() {
        return roll(12);
    }

    public static int d20() {
        return roll(20);
    }
}

When manually testing these projects, you will now have to enter a "seed" value for the random number generator when you begin running your program. The random numbers generated by the dice functions depend on this seed, and reusing the same seed will produce the same sequence of "random" numbers. This allows your program to be tested against specific expected output on projects that rely on some random elements.

You may also find a consistent sequence of numbers instead of random values helpful for manual testing. If you would like to use a random seed, then you can always just press enter without entering a number to let the program run with an arbitrary seed.

Manual Testing

Before using the test script, it's important to test your program manually. The script is only going to help you find mistakes in your code; it won't help you fix them. If your code crashes, contains syntax errors, or has obvious mistakes, then it will be easier to identify and fix them with manual testing. The test script will not function correctly if your program can't run without errors.

Run your program with the java command and provide reasonable input. For example, you could run project 1 with the command java --enable-preview CharacterSheet1.java. If your program crashes, if Java reports syntax errors, or if there are obvious mistakes in the output, then you should fix those before using the test script.

Using the Test Script

To test your program with all of the I/O samples, use the command ./io_test X all, where X is the project's number. For example, ./io_test 1 all would run the test for Project 1. The script should report any I/O samples that your program failed.

If your program fails an I/O sample, then you can check that specific sample with ./io_test X Y, where Y is that sample's number. For example, ./io_test 1 3 would check the third I/O sample for Project 1. The test script will provide additional information about why your program failed for that I/O sample, and this may help you identify the problem.

Running I/O Samples Without the Test Script

The test script only shows which lines from the expected output were missing from your program's output. You can check your program's full output for that I/O sample by using input redirection. For example, to run Project 2 with its seventh sample input, you would use the command java --enable-preview CharacterSheet2.java < io/2/in7.txt. You can open the files io/2/in7.txt and io/2/out7.txt to view the input and expected output for that sample.

Your program passes the test for an I/O sample if every line of the expected output appears in your program's output in the same order and on its own line. This means that any extra output your program produces will not affect the test as long as it appears on a separate line from the required output. This also means that if your output is in the wrong order, then some of it may register as missing. The test should also ignore leading and trailing whitespace in your output.

Tricking the Test Script

It's very easy to trick the test script into passing your code. I will still look at submitted code to ensure it is not plagiarized and that it actually attempts to complete the assignment as directed. If you try to fool the test script instead of completing the assignment, then this will be treated as a violation of the school's academic integrity policy.

As an example, the following program would pass every test if you copied all of the text from the output files into the print statement:

void main() {

println("""
This is a multiline string literal. You can copy as much text here as you want,
and you can include line breaks without \n. If you copy all of the text from the
output files, then this program will print that text and fool the test script.

The script is not hard to fool, but code designed to fool the script is also
easy to spot and will not lead to you getting credit for the assignment.
