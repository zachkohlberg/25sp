Interfaces

An interface is like a fully abstract class1. All methods declared in an interface are implicitly public and abstract. Here are two example interfaces:

public interface Movable {
    // translates an object's location by (dx, dy)
    void move(int dx, int dy);
}

public interface Colorable {
    // changes an object's color
    void setColor(java.awt.Color color);
}

Interfaces allow for something close to multiple inheritance: each class extends exactly one other class, but it can also implement2 any number of interfaces. A concrete class implementing an interface inherits and must override the interface's abstract methods. For example:

import java.awt.Color;

public class SmileyFace implements Movable, Colorable {
    @Override
    public void move(int dx, int dy) {
        this.setLocation(this.getXLocation() + dx, this.getYLocation() + dy);
    }

    @Override
    public void setColor(Color newColor) {
        this.face.setColor(newColor);
    }

    // remaining SmileyFace implementation elided
}

Like the abstract methods in an abstract class, an interface is a way of requiring a class to implement one or more methods. However, this doesn't reuse code; you inherit abstract methods and must write them yourself in each class that implements the interface. If they don't let us reuse code, then why would we use them? Polymorphism

When you declare a variable, the variable is assigned a type. You can either explicitly state the variable's type or use var to have the type inferred from the initial value. The type assigned during declaration is the declared type. However, a variable is not restricted to storing values of its declared type. If the declared type is a class, then it can also store instances of any subclass of its declared type. If the declared type is an interface, then it can also store instances of any class that implements the declared type. This is polymorphism.

Here's an example using inheritance. Irrelevant code will be left out so we can focus on polymorphism.

public class Pet {
    public void speak() {
        // speak implementation elided
    }
    // remaining Pet implementation elided
}

public class Dog extends Pet {
    // Dog implementation elided
}

public class Cat extends Pet {
    // Cat implementation elided
}

public class Main {
    // both parameters have declared type Pet
    public static void haveConversation(Pet a, Pet b) {
        // we can call speak() because it's in the declared type Pet
        a.speak();
        b.speak();
    }

    public static void main(String[] args) {
        Dog dog = new Dog("Fido");
        Cat cat = new Cat("Whiskers");

        // passing instances of Dog to parameters with declared type Pet
        haveConversation(dog, dog);
        // passing instances of Dog and Cat to parameters with declared type Pet
        haveConversation(dog, cat);
        // passing instances of Cat and Dog to parameters with declared type Pet
        haveConversation(cat, dog);
        // passing instances of Cat to parameters with declared type Pet
        haveConversation(cat, cat);
    }
}

The haveConversation method takes Pet parameters, and because Dog and Cat are subclasses of Pet, we can supply instances of those classes anywhere our code expects a Pet, such as arguments for the a and b parameters of haveConversation.

If Java did not support polymorphism, then we would have to write multiple versions of our haveConversation method to allow each of the calls shown above:

public static void haveConversation(Dog a, Dog b) {
    a.speak();
    b.speak();
}
public static void haveConversation(Dog a, Cat b) {
    a.speak();
    b.speak();
}
public static void haveConversation(Cat a, Dog b) {
    a.speak();
    b.speak();
}
public static void haveConversation(Cat a, Cat b) {
    a.speak();
    b.speak();
}

We can also change the previous example to use an interface:

public interface Speaker {
    public void speak();
}

public class Dog implements Speaker {
    // Dog implementation elided
}

public class Cat implements Speaker {
    // Cat implementation elided
}

public class Main {
    // both parameters have declared type Speaker
    public static void haveConversation(Speaker a, Speaker b) {
        // we can call speak() because it's in the declared type Speaker
        a.speak();
        b.speak();
    }

    // the same main method from the first example would also work here
}

This is where interfaces are superior to inheritance: a class can implement any number of interfaces, but it can only extend one other class. We can opt into whatever interfaces we want without committing our class to a specific part of an inheritance hierarchy, and this makes instances of our class compatible with any code that requires one of those interfaces.

Polymorphism does come with a significant restriction: a variable can only be used to call the methods present in its declared type, even if it contains an object with other methods.

Footnotes

1

Interfaces can include default implementations of methods, which starts to blur the line between interfaces and abstract classes. Right now we're more interested in the contract created by an interface and how it relates to polymorphism, so we won't be discussing default interface methods yet.

2

Implementing an interface is like extending a class, and we use the keyword implements to state this relationship. However, the term "implementation" can be used more generally to refer to the code inside of a class or method. The term "interface" can also be used more generally to refer to the public facing portion of a class or library that allows other code to interact with it.
